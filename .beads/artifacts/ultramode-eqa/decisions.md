# Decisions: ultramode-eqa

## Decision Log

| # | Decision | Rationale | Confidence |
|---|----------|-----------|------------|
| 1 | Use native `AbortController` + `signal` parameter on `complete()`/`completeSimple()` instead of `Promise.race` | `StreamOptions` already defines `signal?: AbortSignal` at `@oh-my-pi/pi-ai/dist/types/types.d.ts:143`. Both `complete()` and `completeSimple()` (via `SimpleStreamOptions extends Omit<StreamOptions, "apiKey">`) inherit this field. Using the native signal cancels the underlying HTTP request at the provider level, not just the JavaScript promise. `Promise.race` would leave the HTTP connection open. This is the idiomatic, library-blessed approach. | High |
| 2 | Add optional `timeoutMs: number = DECISION_TIMEOUT_MS` parameter to `decide()` instead of using mutable `let` export or fake timers | `let` exports are live bindings in ES modules — reassigning the local binding does not update importers. Bun's `bun:test` does not provide Jest-style fake timers (`useFakeTimers`). An optional parameter with a production default is standard TypeScript practice, adds zero complexity to production callers (they don't pass the arg), and makes the timeout directly testable with a small value (e.g. 50ms). This matches the existing codebase pattern of exporting internals for testability (`getState`, `runSelection`, `handleTurnEnd` exported for testing — conventions.md line 34). | High |
| 3 | On timeout during `complete()`, do NOT retry with `completeSimple()` — propagate the timeout error immediately | The existing code falls back from `complete()` to `completeSimple()` when `complete()` throws. But if `complete()` threw because the abort signal fired (timeout), retrying with `completeSimple()` would hang for another full timeout window — the provider is unresponsive, not misconfigured. The catch block checks `controller.signal.aborted` and throws immediately, skipping the fallback. If `complete()` throws for a non-timeout reason (e.g. "stream provider failed"), the fallback proceeds as before, with the same signal attached. | High |
| 4 | Use `finally { clearTimeout(timer) }` for timer cleanup instead of manual `clearTimeout` calls at each exit point | The `finally` block runs on all exit paths: success, error, timeout. This is idempotent (`clearTimeout` on an already-cleared timer is a no-op) and avoids duplicating cleanup logic across 3+ exit branches. It prevents timer leaks that would keep the test process alive. | High |
| 5 | Hardcode `DECISION_TIMEOUT_MS = 120_000` as a constant, no env var or user config | 120s is generous for a JSON decision prompt (the model returns a single JSON object). If a model takes >120s, it is hung, not thinking. Configurability is YAGNI — if users need a different value, a separate bead can add it. This follows the existing pattern of `MAX_RETRIES = 3` (hardcoded, not configurable). | High |
| 6 | Place the timeout constant near `MAX_RETRIES` at `index.ts:110` | `MAX_RETRIES` is the existing module-level constant for loop safety limits. `DECISION_TIMEOUT_MS` is the same category — a safety guard for the autonomous loop. Co-locating them makes the safety configuration easy to find and consistent with existing structure. | High |

## Rejected Alternatives

| # | Alternative | Why Rejected | Risk if Re-introduced |
|---|-------------|--------------|----------------------|
| 1 | `Promise.race` between `complete()` and a timeout promise | `Promise.race` would resolve/reject the `decide()` promise on timeout, but the underlying `complete()` HTTP request would continue running — wasting resources and potentially completing later with no consumer. The native `signal` parameter cancels the HTTP request at the provider level. | A latent HTTP connection that completes after timeout could cause unexpected side effects if it triggers callbacks or writes to streams that are still referenced. |
| 2 | Mutable `let DECISION_TIMEOUT_MS` export that tests override | ES module live bindings: reassigning a `let` in the module does not update the value seen by importers (importers get the original binding). Tests importing `DECISION_TIMEOUT_MS` would see the original 120000, not the test override. This is a known JS gotcha that would cause tests to wait 120s or fail silently. | Tests would either hang for 120s (making the suite impractically slow) or pass for the wrong reason (testing with the real timeout instead of a short one). |
| 3 | Bun fake timers (`bun:test` timer mocking) | `bun:test` does not provide a fake timer API like Jest's `jest.useFakeTimers()`. Bun's `mock` module is for module mocking, not timer mocking. This approach is not available. | N/A — the API does not exist. |
| 4 | Export a `setDecisionTimeout(ms)` setter function for tests | Adds a module-level mutable state variable and a setter function — more complexity than an optional parameter. The setter pattern is a less idiomatic testing approach compared to a parameter default. It also introduces test ordering dependencies (one test's setter call could affect another). | Mutable global state between tests — test isolation breaks. A test that sets a short timeout and forgets to restore it would cause all subsequent tests to use the wrong timeout. |
| 5 | No `timeoutMs` parameter — tests wait the full 120s | Tests would take 120+ seconds per timeout test case, making the suite impractically slow. The test suite currently runs in ~1s. Adding 120s per timeout test would make it unusable for rapid iteration. | Test suite becomes unusably slow — developers skip running tests, regressions slip through. |
| 6 | Separate `test/decide-timeout.test.ts` file | The existing convention is that `decide()` error path tests live in `test/error-paths.test.ts` (no-model, no-api-key, completeSimple fallback). A new file would fragment the testing of `decide()` error paths across two files for no structural benefit. The existing file is the right home. | `decide()` error path tests split across two files — harder to find all timeout/error tests for a reader investigating a `decide()` bug. |

## Assumptions

| # | Assumption | Validation | Invalidation Impact |
|---|------------|------------|---------------------|
| 1 | `AbortController` is available in the Bun runtime | Bun is a JavaScript runtime built on JavaScriptCore. `AbortController` is a Web standard available in all modern JS runtimes (Node 15+, Bun, Deno, browsers). Verified: no polyfill needed. | If `AbortController` is not available, the code would throw `ReferenceError` at runtime. Mitigation: the build step (`bun build index.ts --no-bundle`) would catch this at type-check time if the type is missing. |
| 2 | `complete()` and `completeSimple()` respect the `signal` parameter and abort the HTTP request when the signal fires | The `signal?: AbortSignal` field is declared in `StreamOptions` (types.d.ts:143) and inherited by `SimpleStreamOptions`. The type exists and is typed as `AbortSignal`. Verified via type definitions. Runtime behavior (whether the provider's HTTP client actually checks the signal) is provider-dependent, but the loop's correctness only requires `decide()` to reject on timeout — which it does via the `setTimeout` → `controller.abort()` → signal check in the catch block. | If a provider ignores the signal, the HTTP request continues in the background. `decide()` still returns on timeout (the promise rejects). The only impact is a wasted HTTP connection — no correctness issue. |
| 3 | The existing caller try/catch blocks in `runSelection()` and `handleTurnEnd()` will catch the timeout error | Verified by reading index.ts:436-446 and index.ts:599-610. Both callers wrap `decide()` in try/catch, notify the user, set `state.mode = "idle"`, persist state, and return. The timeout error is a standard `Error` instance — it will be caught. | If a caller does NOT catch the error, the unhandled rejection would crash the event handler. Verified: both callers have try/catch. No risk. |
| 4 | `setTimeout` and `clearTimeout` are available in the OMP extension runtime | These are global functions available in all JavaScript runtimes. The OMP extension runs in Bun, which provides standard global timers. | If timers are not available, the code would throw `ReferenceError`. Mitigation: the build step catches this. |
| 5 | The `timeoutMs` parameter with a default value does not break existing callers | Both callers (`runSelection` at index.ts:437, `handleTurnEnd` at index.ts:601) call `decide(ctx, prompt)` with two arguments. Adding a third optional parameter with a default value is backward-compatible in TypeScript — existing two-arg calls continue to work. | If a caller passes additional arguments positionally, the parameter shift could cause issues. Verified: both callers use exactly two arguments. No risk. |
| 6 | 120s is a reasonable timeout for all supported models | The decision prompt is short (a few hundred tokens). The expected response is a single JSON object. Reasoning-capable models should respond within 10-30s. 120s provides a 4-12x safety margin. If a model takes >120s for a short prompt, it is likely hung. | If a legitimate slow model (e.g. o1 with deep reasoning) takes >120s, the loop would incorrectly idle. The user would need to re-engage with `/ultramode on`. This is an acceptable tradeoff — false positives on very slow models are better than infinite hangs. |
