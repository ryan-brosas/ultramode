# Decisions: ultramode-yk6

## Decision Log

| # | Decision | Rationale | Confidence |
|---|----------|-----------|------------|
| 1 | Use `bun build index.ts --target bun --external @oh-my-pi/pi-ai --external @oh-my-pi/pi-coding-agent` for the build step | `--no-bundle` fails because it can't resolve `@oh-my-pi/pi-ai` (OMP global package, no local node_modules). `--target bun` without `--external` works locally but fails in CI (no OMP global packages). `--target bun --external` works in both environments ‚Äî verified in clean CI simulation (`/tmp/ci-test` with `env -u BUN_INSTALL HOME=/tmp/fakehome`, exit 0, 19,623 bytes stdout). The `--external` flags are correct because these packages are truly external (provided by OMP at runtime). | High |
| 2 | Use `bun test test/` for the test step, not `bun test` or `bun test test/**.test.ts` | `bun test test/` is the existing convention (test/run.sh uses it, all prior beads reference it). It's explicit about the test directory. `bun test` without a path would also work but is less explicit. `bun test test/**.test.ts` is more restrictive than needed ‚Äî any new test file in test/ should be picked up automatically. | High |
| 3 | Remove the `clean` script entirely, not replace it with `rm -rf dist/` | The build step writes to stdout (verified ‚Äî Finding 7), not to a file. No `dist/` directory is created. There's nothing to clean. Adding a `clean` script would be speculative (YAGNI). If a future bead adds file output to the build, it can add a clean script then. | High |
| 4 | Use `bun-version: latest` in setup-bun, not a pinned version | OMP itself doesn't pin a Bun version (no `engines` field in package.json). Bun 1.x is backward compatible. Using `latest` ensures CI runs on the same Bun version the developer is likely using locally. Pinning to a specific version is premature optimization ‚Äî if reproducibility issues arise, pin then. | Medium |
| 5 | Trigger CI on both `push` and `pull_request`, both scoped to `main` | `push` catches direct commits to main (e.g., hotfixes). `pull_request` catches PRs before merge. Scoping to `main` avoids duplicate runs on feature branches (the PR already triggers CI). This is the standard pattern for small projects. | High |
| 6 | No dependency caching step in CI | The project has zero npm dependencies (no `dependencies`, no `devDependencies` in package.json). There's no `node_modules` to cache. `bun test` uses only Bun builtins and `mock.module` intercepts. `bun build --target bun --external` doesn't need packages installed. Adding caching would be pure overhead. | High |
| 7 | No type checking (`tsc --noEmit`) in CI | Requires OMP global packages (`@oh-my-pi/pi-ai`, `@oh-my-pi/pi-coding-agent` type definitions) not available in CI. Even with symlinks, has 2 pre-existing type errors at index.ts:376,379 (`Model<Api> \| undefined`). `bun build --external` provides syntax/import verification without type checking. Type checking is a separate bead after fixing the pre-existing errors. | High |
| 8 | No linting in CI | No linter is configured (no `.eslintrc`, no `biome.json`, no `deno.json`). Adding linting requires choosing a linter, configuring rules, and fixing violations ‚Äî a separate concern. YAGNI; the test suite catches behavioral regressions. | High |
| 9 | No `.gitignore` changes | The build step writes to stdout, not to a file (verified ‚Äî Finding 7). No `dist/` directory is created. There's nothing to gitignore. This was initially assumed to require a `dist/` entry, but investigation proved otherwise. | High |
| 10 | Single job (`ci`), not multiple jobs (test, build) | The test suite runs in 0.32s and the build in ~0.5s. Splitting into separate jobs would add ~15s of runner setup overhead per job (checkout + setup-bun) for no benefit. A single job with sequential steps is faster and simpler. If test or build times grow significantly, splitting becomes worthwhile. | High |

## Rejected Alternatives

| # | Alternative | Why Rejected | Risk if Re-introduced |
|---|-------------|--------------|----------------------|
| 1 | `bun build index.ts --no-bundle` for the build step | Fails: `error: Could not resolve: "@oh-my-pi/pi-ai". Maybe you need to "bun install"?` ‚Äî `--no-bundle` still needs to resolve imports but can't find OMP global packages (Finding 1). | CI build step would fail immediately. A developer trying this would see the error and need to debug the OMP global package resolution issue. |
| 2 | `bun build index.ts --target bun` (without `--external`) | Works locally (resolves from `~/.bun/install/global/node_modules/@oh-my-pi/`) but fails in CI (no OMP installed). The 7.7MB bundle includes all of pi-ai and its dependencies, which is unnecessary ‚Äî we only need syntax verification. | CI build step would fail with the same "Could not resolve" error. A developer testing locally would see it work and be surprised when CI fails. |
| 3 | `tsc --noEmit` for type checking | Requires OMP global packages for type definitions. Has 2 pre-existing type errors at index.ts:376,379 (`Model<Api> \| undefined`). Would need either OMP installed in CI (heavyweight) or CI-specific symlinks (fragile). | CI would fail on the pre-existing type errors, blocking all PRs until those errors are fixed. Those errors are in `decide()` (ultramode-eqa's domain) and should be fixed in a separate bead. |
| 4 | `npm install` + dependency caching in CI | The project has zero npm dependencies. No `node_modules` to install or cache. Adding an install step would be a no-op that adds time. | Adds ~5-10s of unnecessary CI time. A developer might think the project has dependencies and be confused by the empty install. |
| 5 | Matrix testing across multiple Bun versions | OMP targets a specific Bun version (the one installed with OMP). Testing against multiple versions would catch version-specific regressions, but OMP itself doesn't do this. YAGNI ‚Äî if a version-specific issue arises, add matrix testing then. | Adds CI complexity and time. A failing matrix test on an unsupported Bun version would block PRs. |
| 6 | Separate `test` and `build` jobs | The test suite (0.32s) and build (0.5s) are fast enough that splitting into separate jobs adds more overhead (duplicate checkout + setup-bun, ~15s each) than it saves. | CI time would roughly double for no benefit. A developer would wait longer for CI to complete. |
| 7 | Pin `bun-version` to a specific version (e.g., `1.3.14`) | Premature optimization. Bun 1.x is backward compatible. OMP doesn't pin a version. If a version-specific issue arises, pin then. | CI might break if `latest` introduces a breaking change. Mitigation: if this happens, pin immediately. Low likelihood ‚Äî Bun 1.x has been stable. |
| 8 | Add a `clean` script (`rm -rf dist/`) | The build step writes to stdout, not to a file. No `dist/` is created (Finding 7). There's nothing to clean. | A developer might add a build step with file output later and not realize there's no clean script. But that's a separate concern ‚Äî YAGNI. |
| 9 | Add `dist/` to `.gitignore` | No `dist/` directory is created by the build step (Finding 7). Adding a gitignore entry for a nonexistent directory is harmless but unnecessary. | None ‚Äî it's a no-op. But it adds noise to the diff and the .gitignore file without value. |
| 10 | Use `actions/setup-node@v4` instead of `oven-sh/setup-bun@v2` | The project uses Bun, not Node directly. `setup-node` doesn't install Bun. `oven-sh/setup-bun@v2` is the official Bun setup action maintained by the Bun team. | CI would fail because `bun` command wouldn't be available. |
