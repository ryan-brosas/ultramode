# Decisions: ultramode-fpj

## Decision Log

| # | Decision | Rationale | Confidence |
|---|----------|-----------|------------|
| 1 | Build as an OMP extension (`.ts` default export), not an external daemon process | Extensions run in-process with access to `turn_end`, `sendUserMessage`, `runEphemeralTurn`, `appendCustomEntry`, `ctx.ui.notify`, and `ctx.ui.setWidget`. An external daemon would lose all of this and require JSON parsing of `omp -p --mode json` output. The `autoresearch` built-in extension proves the loop pattern works end-to-end. | High |
| 2 | Use `runEphemeralTurn` for decisions instead of mechanical string matching | The user explicitly asked for LLM-driven senior-engineer decisions, not automation. `runEphemeralTurn` provides a pure LLM call with `toolChoice: "none"` that doesn't pollute the conversation or execute tools. This is the "senior engineer brain" — it decides what to work on, whether to proceed, and whether to retry. | High |
| 3 | Use `sendUserMessage` for phase chaining on `turn_end` | `sendUserMessage` injects a user message that the agent processes as if the user typed it. This means the template's command interception (`native-command-override.ts` for `/plan` and `/review`) works normally. The agent receives the full command body and executes it. | High |
| 4 | Stop at `/pr`, never run `/close` | RULE #6 ("the human always gets the last call on merges") is a system-prompt-level rule. The `/close` command has a mandatory human approval gate for memory-file updates (`close.md:99-117`). Auto-merging or auto-closing violates both. The extension stops at `/pr`; the human merges and runs `/close`. | High |
| 5 | Persist state via `appendCustomEntry("ultramode-control", state)` | This is the exact pattern `autoresearch` uses (`appendEntry("autoresearch-control", {mode, goal})`). Custom entries survive session restart and are reconstructable via `getBranch()`. No external state file needed. | High |
| 6 | Parse JSON from `runEphemeralTurn` free text with fail-safe to "stop" | `runEphemeralTurn` does not support a `schema` parameter for structured output. The extension extracts the first `{...}` block from the reply text, parses it, and validates the `action` field. If parsing or validation fails, the loop stops and notifies the user — never proceeds blindly with an ambiguous action. | Medium |
| 7 | Hard retry cap: 3 attempts per phase | Prevents infinite loops on failing beads. After 3 retries, the bead is marked blocked via `br update --status blocked` and the extension picks the next ready bead. This matches the template's existing guardrail ("Max 3 fix cycles"). | High |
| 8 | Use `pi.exec` to call `br` and `bv` commands | The existing `workflow-gate.ts` extension demonstrates `pi.exec("br", [...])` for executing br commands from extension context. The `ultramode` extension reuses this pattern for `bv --robot-triage --format json` and `br scheduler --json`. | High |
| 9 | Register `/ultramode` command with subcommands (on/off/status/continue) | The `autoresearch` extension registers `/autoresearch` via `pi.registerCommand`. The `ultramode` extension follows the same pattern. Subcommands are parsed from the args string. | High |
| 10 | Two-tier architecture: decision agent + executor agent | The decision agent (`runEphemeralTurn`) thinks — it decides what to work on and whether to proceed. The executor agent (main session) builds — it receives phase commands via `sendUserMessage` and executes them with the template's governance (workflow-gate, review agents, evidence). This separation prevents the decision logic from being entangled with implementation. | High |
| 11 | Ship as a standalone plugin repo (`ultramode`), not in the omp-template | The omp-template is a template — it shouldn't ship a specific autonomous loop. The extension is a standalone package installable via `omp install`, usable across any project that uses the template. Modeled on `omp-makora-provider` package structure. | High |

## Rejected Alternatives

| # | Alternative | Why Rejected | Risk if Re-introduced |
|---|-------------|---------------|----------------------|
| 1 | External daemon process spawning `omp -p --mode json` per phase | Loses in-process state, requires JSON parsing of undocumented output format, no tool-call interception, no UI feedback, no `turn_end` events, no `runEphemeralTurn`. The extension wins on every axis. | Re-introducing a daemon would duplicate the extension's functionality with worse ergonomics. It would need its own session management, output parsing, and error recovery — all of which the extension gets for free from the harness. |
| 2 | Mechanical phase chaining (string-match "Next:" markers in agent output) | Not agentic. Picks top bv recommendation mechanically, no judgment about what's worth doing. The user explicitly asked for LLM-based senior-engineer decisions. | Mechanical chaining would produce PRs for low-value work, miss root-cause diagnosis, and retry failures without understanding why. It's automation, not engineering. |
| 3 | Modify the template commands to auto-advance phases | Violates the "template unchanged" principle. Commands should remain single-turn and human-usable. The loop logic belongs in an extension, not baked into commands. | Modifying commands would couple the autonomous loop to the template, making it impossible to use the template without the loop. It would also make commands harder to debug and test in isolation. |
| 4 | Use the `completion()` eval helper instead of `runEphemeralTurn` | `completion()` is available in the eval tool (Python/JS kernel) but not directly from extension context. `runEphemeralTurn` is the extension-accessible LLM call. Using `completion()` would require shelling out to the eval kernel, which is slower and more complex. | If `runEphemeralTurn` proves inaccessible from extension context (Risk #1), the fallback is `pi.exec("omp", ["-p", "--mode", "json", decisionPrompt])` as a subprocess, not `completion()`. |
| 5 | Build into the omp-template repo | The template is a template — it should provide governance infrastructure, not opinionated autonomous loops. Different users may want different autonomous strategies. | Bundling the extension with the template would force all template users to either use this specific loop or disable it. Keeping it separate lets users choose. |
| 6 | Support `schema` parameter on `runEphemeralTurn` for structured output | `runEphemeralTurn` does not support `schema` — it returns free text. Adding schema support would require modifying the OMP harness, which is out of scope. | If OMP adds schema support to `runEphemeralTurn` in the future, the extension should adopt it to eliminate JSON parsing failures. This is a future enhancement, not a current requirement. |

## Assumptions

| # | Assumption | Validation | Invalidation Impact |
|---|------------|------------|---------------------|
| 1 | `ctx.session.runEphemeralTurn` is accessible from extension event handlers | Verified via `strings` analysis of the OMP binary — the method exists on `AgentSession` and is exposed via `ctx.session`. The `hello-extension` and `autoresearch` examples confirm `ctx.session` is available in handler contexts. | If inaccessible, the extension falls back to `pi.exec("omp", ["-p", "--mode", "json", prompt])` as a subprocess for decisions. Less efficient and loses conversation context, but functional. |
| 2 | `ctx.session.sendUserMessage` can be called from `turn_end` handler without deadlock | The `autoresearch` extension chains messages after turn events using the same pattern. `sendUserMessage` is designed for extension use (confirmed: `sendUserMessage: this.#sendExtensionUserMessage`). | If deadlock occurs, defer via `queueMicrotask(() => ctx.session.sendUserMessage(...))` to break re-entrancy. If that also deadlocks, the extension cannot chain phases and the approach fails. |
| 3 | The session's configured model is reasoning-capable | The extension uses the session's model for `runEphemeralTurn`. If the model is a cheap/fast model, decision quality suffers. | Document in README that the session model should be reasoning-capable. The extension cannot override model selection — that's the user's responsibility. |
| 4 | `br scheduler --json` exists and returns ready work | Verified in `br/SKILL.md:64` — `br scheduler --json` returns evidence-ranked ready work. Fallback: `br list --status open --json` if scheduler is unavailable. | If `br scheduler` doesn't exist, the extension uses `br list --status open --json` and lets the LLM decide which bead to work on from the list. |
| 5 | `bv --robot-triage --format json` returns structured triage output | Verified in `bv/SKILL.md:54-68` — returns recommendations, quick_wins, blockers, project_health. | If `bv` is unavailable, the decision prompt includes "bv unavailable" and the LLM decides based on `br list` output alone. |
| 6 | Multiple `tool_call` handlers can coexist (ultramode + workflow-gate) | OMP's extension runner allows multiple handlers for the same event — they are additive. Both handlers run; each can independently block. | If OMP does not support multiple handlers, the ultramode handler would need to be merged with workflow-gate, increasing coupling. Low likelihood based on the extension runner architecture. |
| 7 | `appendCustomEntry` entries survive session restart and are reconstructable via `getBranch()` | Confirmed in `autoresearch` extension: `reconstructControlState(ctx.sessionManager.getBranch())` scans entries for `autoresearch-control` custom type. | If entries don't survive restart, state is lost on crash. The extension would default to `mode: "off"` and notify the user to run `/ultramode on`. Functional but not seamless. |
